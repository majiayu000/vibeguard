# Product Spec — prompt 注入规则的 with/without 配对评测（副作用门）

## Linked Issue

GH-686

complexity: large

## 用户问题

行为评测门（`eval/behavior/`，#684 扩展）只覆盖 **hook 强制**的规则：它能确定性地
同时检出漏拦和误拦。**prompt 注入规则**（compact L1-L7 表 + `rules/claude-rules/`
下的原生规则文件）没有等价的副作用门。

一条 prompt 规则可以在它自己的目标场景上"正确"，同时**在无关任务上劣化 agent
行为**——约束衰减 / 注意力挤占（U-32 引用的研究）。今天没有任何东西在规则上线前
度量这一点。VibeGuard 自己的 U-32 规定活跃约束 ≤15，但没有实验数据支撑任何一条
新规则是否值得占用这个预算。

现状（本会话核实）：

- `eval/run_eval.py:215` 已经把规则来源参数化，并对规则文本取 `rule_digest`。
- 但规则文本有**两个**注入来源：`--rules-dir` 的规则树，以及无条件拼上的
  `--core-rules-file`（默认 `claude-md/vibeguard-rules.md`，其中以表格行重复规则正文）。
- `eval/datasets/v1.jsonl` 共 40 条样本，`rule == "NONE"` 仅 4 条，单个规则最多 4 条。
- 整个流程是**单次运行**：只有"带全部规则跑一遍"，没有"不带候选规则跑一遍"
  作为对照，因此无法归因任何行为差异。

## 目标

- 提供配对（with / without 候选规则）运行能力，复用既有的 model-backed 路径。
- 规则通过门需要**同时**满足两条证据轴：
  - **目标场景改善**：这条规则存在的理由所对应的行为，确实发生了改变。
  - **非目标不回归**：一组与该规则无关的固定任务没有变差。具体口径取决于 D2：复用
    现有 grader 时它是"误报率不上升"，引入 pairwise judge 时才是"质量没有下降"。
- 产出可复现的差异指标，并作为规则 PR 模板中的必需证据。

## 非目标

- 不替换确定性行为门。hook 强制的规则继续由 `eval/behavior/` 覆盖。
- 不做上线后的持续追踪。持续精度归 guard-precision-tracker（GH-675）。
- 不在本 issue 中新增或修改任何实际规则。
- 不把配对评测接入默认 CI 阻断路径：它需要真实模型调用，成本与不确定性都不适合
  每次 push 运行。它是规则 PR 的按需门。

## Behavior Invariants

1. B-001: 必须能在一次调用中，对同一样本集执行 with-candidate 与 without-candidate
   两次运行，且两次运行除候选规则文本外的所有输入（模型、样本集、其余规则）完全一致。
2. B-002: 两次运行的输入身份必须可审计：分别记录 `rule_digest`、
   `dataset_digest`、`sample_set_digest` 与解析后的模型 ID；除 `rule_digest` 外必须相等，
   不相等时以错误终止而不是继续比较。
3. B-003: 候选规则的排除必须覆盖**全部**注入来源，并以差集方式验证：with 文本剔除
   候选规则后必须与 without 文本逐字节相等，且 without 文本中不得出现候选 ID 的任何
   独立 token。只检查"标题行不在"不足以证明对照有效。
4. B-004: 必须区分两组样本：**目标样本**（`rule` 字段**精确等于**候选规则 ID）与
   **非目标样本**（一组与候选规则无关的固定集合）。两组都为空时以错误终止。前缀匹配
   或混入通用样本都会污染目标轴。
5. B-005: 判定必须同时要求两条轴通过；任一条不通过即整体不通过。单轴通过不得被
   报告为"通过"。
6. B-006: 必须区分"没有回归"与"样本量不足以判断"。**任一轴**样本量低于配置的最小值
   时，结果为 `inconclusive`，不得报告为 `pass`。
7. B-007: 任何跳过的样本（API 失败、解析失败）必须计入并可见；被跳过的样本不得
   静默地从分母中消失。跳过率超过阈值、或 with 与 without 的跳过率之差超过阈值时，
   结果为 `inconclusive` —— 跳过率偏差会直接主导两轴的差值。
8. B-008: 必须提供离线 dry-run 路径：不调用模型，输出两次运行的输入身份与样本
   划分，供确定性测试与人工核对使用。
9. B-009: 规则 PR 模板必须要求附上配对评测结果，或显式声明豁免理由；豁免仅适用于
   非 prompt 注入类改动，且需要维护者批准记录。无约束的豁免等于没有门。
10. B-010: 阈值未标定时，整体判定强制为 `inconclusive`，永不输出 `pass`。仅打印警告
    不足以阻止下游把未标定的门当作已标定的门引用。
11. B-011: `inconclusive` 与不通过一样以非零退出码结束。退出码为 0 的 `inconclusive`
    等于门不存在。

## 验收标准

- [ ] 存在配对运行入口，接受候选规则 ID 与非目标样本集。
- [ ] 候选规则从两个注入来源同时剔除，且由差集断言证明；整文件删除必须被拒。
- [ ] 输入身份不一致时以非零退出码终止并说明哪一项不一致。
- [ ] 目标轴与非目标轴分别报告，整体判定为两者的合取。
- [ ] 任一轴样本量不足、跳过率超阈值、或阈值未标定时报告 `inconclusive`。
- [ ] `inconclusive` 与不通过均以非零退出码结束。
- [ ] dry-run 路径不需要 API 密钥即可运行，并被确定性测试覆盖。
- [ ] 规则 PR 模板包含配对评测证据要求，且豁免受限。

## 边界情况清单

| 类别 | 判定（covered: B-xxx / N/A + 原因） |
| --- | --- |
| 空/缺失输入 | covered: B-004；两组样本都为空时终止 |
| 错误与失败路径 | covered: B-002, B-007；输入漂移终止，跳过样本可见 |
| 授权/权限 | covered: B-008；无密钥时 dry-run 仍可用，真实运行明确报缺密钥 |
| 并发/竞态 | N/A — 两次运行串行，无共享可写状态 |
| 重试/幂等 | covered: B-002；同一输入身份可复现，摘要用于比对 |
| 非法状态转换 | covered: B-005；单轴通过不得升格为整体通过 |
| 兼容/迁移 | covered: B-001；共享 `run_eval` 的下层组件，不改单次运行语义 |
| 降级/回退 | covered: B-006, B-010, B-011；样本不足与未标定都是 `inconclusive` 而非 `pass`，且非零退出 |
| 证据与审计完整性 | covered: B-002, B-003；候选规则的剔除由差集断言证明，假对照会被拒 |
| 取消/中断 | covered: B-007；中断导致的未完成样本按跳过计入 |

## 开放问题

初版把三个问题都推给维护者。核对代码后，其中两个 spec 自己就能定：

- **非目标任务集从哪来 → 已定：必须新建。** `eval/datasets/v1.jsonl` 只有 4 条
  `NONE` 样本，而非目标轴需要的下限是 30。从 v1 划分在算术上不成立。
- **样本量标定 → 已定：先落地方向性门。** `calibrated: false` 强制 `inconclusive`
  之后，未标定的门不会产生误导性的 `pass`，标定实验可以独立进行。

**仍需维护者裁定：打分方式。** `eval/dataset.py:105-118` 强制 `fp` 样本必须
`rule="NONE"` + `expected_action="allow"`，现有 grader 只产出 detection rate 与
false-positive rate。复用它时非目标轴度量的是**误报漂移**，不是 issue 说的"质量
没有下降"。要测"质量"必须引入 pairwise judge。这决定数据集形态和打分实现。

## 发布说明

配对评测是按需运行的规则 PR 门，不进默认 CI。它需要 Anthropic API 密钥；
dry-run 路径不需要。
