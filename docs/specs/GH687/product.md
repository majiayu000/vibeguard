# Product Spec — 证据必须可证明被执行（W-21）+ W-01 通道可信性 step 0

## Linked Issue

GH-687

complexity: medium

## 用户问题

长上下文会话在退化后会**编造"决定性证据"**：叙述一个从未发生过的实验，再基于
这个虚构观察构建机制理论。GH-687 记录的真实事故里，一个多小时的 Claude Code
会话连续提出并证伪三个根因理论，最终建议用户禁用无辜的 VibeGuard PostToolUse
hook。事后 grep 该会话的 transcript JSONL 证明：被称为"决定性实验"的
`/tmp/decide.txt` 读写从未出现在任何 `tool_use` / `tool_result` 记录里，只出现
在助手自己的叙述中；被抱怨的"乱码输出"在所有真实 tool_result 中都不存在。

现有规则挡不住这一类失败：

- W-01 / W-03 / W-16 在形式上都被满足了 —— 会话复现了、引用了会话内证据、用了
  新鲜输出，但被引用的证据是会话自己编造的。**引用纪律只校验"证据被引用了"，
  不校验"证据真的发生过"。**
- W-02（3 次失败后回退）从未触发，因为每次证伪都被包装成"离真正根因更近了"，
  而不是"一次失败的修复"。
- 调试协议的第一步就是"调查根因"，没有任何一步先问"我的观察信号本身可信吗"。
  观察通道一旦被污染，下游每个假设都不可验证。

## 目标

- 新增一条严格规则，要求"决定性 / 根因锁定"级别的断言必须能通过**会话外通道**
  （transcript JSONL、磁盘文件、git、退出码、哈希）证明确实执行过。
- 在 W-01 的调试协议前面插入 step 0：命名任何机制之前，先确认观察通道可信。
- 明确"agent 指控 harness / hook / 环境损坏"本身是红旗信号，在用会话外通道证明
  之前不得据此采取行动（尤其是禁用 hook）。
- 给出比 W-02 更紧的长会话止损判据：同一排查中根因理论被证伪 2 次即杀会话重开，
  状态从磁盘产物恢复而不是从退化的上下文恢复。

## 非目标

- 不新增 hook 或运行时的机械拦截。本 issue 交付的是 agent/reviewer 契约层规则，
  与 W-01 / W-02 / W-03 / W-16 同级。
- 不改变 W-02 的 3 次阈值；W-21 的 2 次判据只作用于"根因理论被证伪"这一类事件。
- 不修改默认安装的规则注入体积（GH-687 第 3 节的 U-32 / W-19 自合规提案）。该项
  改变所有用户的安装默认行为，属于架构级决策，单列给维护者定夺。
- 不修改 SEC-13 中关于 `updatedToolOutput` 的既有描述。

## Behavior Invariants

1. B-001: 规范源中必须存在一条 ID 未被占用的新规则，标题为"证据必须可证明被执行"，
   severity 为 strict，并带 `**Compact guidance:**` 行。
2. B-002: 新规则必须要求"决定性 / 根因锁定"级断言至少能被一个会话外通道证明：
   transcript JSONL、文件系统、git，或持久化到磁盘的退出码 / 哈希。
3. B-003: 新规则必须要求优先使用**持久化到磁盘的单值信号**（退出码、哈希）而不是
   多行文本回忆，并说明编造风险随输出长度增长。
4. B-004: 新规则必须把"agent 指控 harness / hook / 环境损坏"标记为红旗，并要求在
   禁用任何 hook 之前先取得会话外证据。
5. B-005: 新规则必须规定会话止损判据：同一排查中根因理论被证伪 2 次 → 停止原地
   迭代、杀掉会话重开、从磁盘产物恢复状态。
6. B-006: W-01 的调试协议必须以 step 0（通道可信性检查）开头：当工具输出自相矛盾
   时，首个假设必须是"我自己的读取 / 上下文退化了"，用会话外通道核验之后才能
   假设文件系统、harness 或 hook 损坏。
7. B-007: 新规则 ID 不得与仓库中任何既有规则 ID 冲突（`W-20` 已被"长任务必须钉住
   运行时"占用）。
8. B-008: 生成的派生文档（`rules/universal.md`、`docs/rule-reference.md` 等）必须与
   规范源一致，由 `scripts/generate_rule_docs.py` 生成而非手写。

## 验收标准

- [ ] 新规则以规范源形式落在 `rules/claude-rules/common/` 下，可被
      `scripts/generate_rule_docs.py` 解析。
- [ ] W-01 的四阶段调试协议前多出 step 0，且 `W-21` 与 `W-01` 互相交叉引用。
- [ ] `python3 scripts/generate_rule_docs.py --check` 通过（派生文档已同步）。
- [ ] `bash scripts/ci/validate-rules.sh` 与
      `bash scripts/ci/validate-generated-rule-docs.sh` 通过。
- [ ] 存在一个确定性测试，断言 B-001 ~ B-007 的规则文本要素存在。

## 边界情况清单

| 类别 | 判定（covered: B-xxx / N/A + 原因） |

| 空/缺失输入 | covered: B-008；缺失派生文档由 `--check` 报错 |
| 错误与失败路径 | covered: B-005；证伪 2 次后必须停止而不是继续迭代 |
| 授权/权限 | covered: B-004；禁用 hook 前必须先有会话外证据 |
| 并发/竞态 | N/A — 本规则是 agent 契约，不涉及并发写入 |
| 重试/幂等 | covered: B-008；生成器对同一规范源输出稳定 |
| 非法状态转换 | covered: B-007；ID 冲突即非法，必须换未占用 ID |
| 兼容/迁移 | covered: B-007；不复用 GH-687 原文提议的 `W-20` |
| 降级/回退 | covered: B-002；会话内证据不足时降级为"未验证"，不得升格为决定性 |
| 证据与审计完整性 | covered: B-002, B-003；这是本规则的核心 |
| 取消/中断 | covered: B-005；杀会话是显式的、被规定的终止路径 |

## 发布说明

已安装的用户需要重新运行 `setup.sh`（或等待下一次规则同步）才能在
`~/.claude/rules/vibeguard/` 拿到新规则文件。规则为 agent/reviewer 契约层，
不改变任何 hook 的拦截行为，不影响既有 CI 结果。
