## 知识摘要

### 1. Harness Engineering 101 — 从提示工程到编排工程的范式演进（14 分）
- **来源**: [AIDailyBrief](https://creators.spotify.com/pod/show/nlw/episodes/Harness-Engineering-101-e3hqs2e)
- **核心**: AI 工程正经历三阶段演进：Prompt Engineering → Context Engineering → Harness Engineering
- **关键发现**:
  - Harness Engineering 不是优化单个提示或上下文，而是构建围绕模型的完整系统基础设施
  - 「所有 AI 产品正在收敛为相同的形状」——成功部署需要类似的支撑架构
  - Anthropic 托管代理公告被解读为 Harness Engineering 的实践典范
  - 模型本身的能力改进在 diminishing returns，系统层面的编排成为新的差异化战场
- **VG 关联**: 与 VibeGuard 的「工作流成熟度阶梯」(Manual → Skill → Automation) 理念一致；Harness = 我们的 hooks + rules + skills 体系

### 2. HiL-Bench：编码代理何时该求助的首个基准（14 分）
- **来源**: [arXiv:2604.09408](https://arxiv.org/abs/2604.09408)
- **核心**: 前沿模型在规格不完整时表现急剧下降，问题不在能力而在判断力
- **关键发现**:
  - 三种失效模式：过度自信无不确定性检测、高不确定性但持续犯错、模糊升级不自纠
  - Ask-F1 指标：防止通过问题轰炸刷分，捕捉过度提问与静默猜测的张力
  - 没有前沿模型能在「是否求助」决策中恢复其完整信息下的性能
  - RL 训练证明判断力可训练，32B 模型在求助质量和任务性能上均有提升且可跨域迁移
  - 模型学会「检测不可解决的不确定性并据此行动」而非记忆特定领域启发式
- **VG 关联**: 直接强化 W-02（连续失败 3 次后退）和 W-10（发布确认规则）；Ask-F1 思路可用于评估 VibeGuard 的 agent 求助行为

### 3. SkillMOO：Agent Skill 的多目标自动优化——少即是多（14 分）
- **来源**: [arXiv:2604.09297](https://arxiv.org/abs/2604.09297)
- **核心**: 自动优化 Skill 包以平衡成功率、成本和运行时间
- **关键发现**:
  - 双代理进化框架：Solver Agent 评估 + Optimizer Agent 分析失败 + NSGA-II 演化选择
  - SkillsBench 结果：通过率提升 131%、成本降低 32%
  - **核心洞察：「有效 Skill 包偏好最小化、聚焦内容而非累积指令」——战略性裁剪 > 不断堆积**
  - 少即是多：最优 Skill 不是「所有好建议的合集」，而是精准选择的最小集合
- **VG 关联**: 直接验证 U-02 精神（不为 1 次代码提取抽象）；提示 VibeGuard 105 条规则可能存在优化空间——裁剪优于累积

### 4. 解剖代理框架的 Bug 触发器与失效模式（14 分）
- **来源**: [arXiv:2604.08906](https://arxiv.org/abs/2604.08906)
- **核心**: 409 个已修复 Bug 的实证研究，覆盖 CrewAI、AutoGen 等 5 个框架
- **关键发现**:
  - 五层结构抽象模型：从编排层到基础设施层
  - 三类特有根因：模型相关故障、认知上下文管理错误、编排故障
  - 特有症状：意外执行序列、用户配置被忽略
  - Bug 触发模式跨框架可迁移——可建立通用测试策略
- **VG 关联**: 「用户配置被忽略」与 U-26（声明-执行完整性）直接对应；「认知上下文管理错误」是 VibeGuard 的核心防御目标

### 5. OpenKedge：用执行绑定安全和证据链治理代理变异（14 分）
- **来源**: [arXiv:2604.08601](https://arxiv.org/abs/2604.08601)
- **核心**: 将代理变异重定义为治理过程而非 API 调用的直接后果
- **关键发现**:
  - 声明式意图提案 → 状态/时间/策略约束评估 → 执行合约（限定动作/资源/时效）
  - 意图到执行证据链 (IEEC)：加密链接意图→上下文→策略→执行→结果，可审计重建
  - 通过临时任务身份（ephemeral task-oriented identities）隔离执行权限
  - 确定性仲裁竞争意图、隔离不安全执行，同时保持高吞吐
- **VG 关联**: IEEC 概念可启发 VibeGuard exec-plan 的证据链设计；执行合约思路与 W-10（四项确认）、W-14（文件所有权）互补

### 6. Sherlock：AI 安全代码审查自动化经验（12 分）
- **来源**: [Sourcegraph Blog](https://sourcegraph.com/blog/lessons-from-building-sherlock-automating-security-code-reviews-with-sourcegraph)
- **核心**: Sourcegraph 安全团队用 LLM 自动化安全代码审查，2 个月扫描 400+ PR
- **关键发现**:
  - 架构：PR 触发 → diff/SAST 告警摄取 → Sourcegraph API 上下文增强 → LLM 定制 Prompt → 扫描器交叉关联降噪 → Slack/SIEM 输出
  - 发现 3 高危 + 4 中危 + 12 边界案例，每天节省 30+ 分钟/安全工程师
  - 降噪关键：定制威胁模型 Prompt + 代码上下文增强 + 扫描器交叉验证
  - 挑战：LLM 标记不存在的漏洞、缺乏代码导航、倾向通用建议而非特定变更分析
- **VG 关联**: Sherlock 的扫描器交叉关联降噪策略可借鉴到 vibeguard:review 和 vibeguard:cross-review 中
