# Execution Pinning Rules

## W-20: Execution pinning for long-running tasks (strict)
Long-running agent tasks must pin the execution environment before work starts. If the runtime, tool surface, or rule set changes mid-task, the task must stop and report runtime drift instead of silently continuing under a different environment.

**Trigger**:
- The task spans 3 or more agent steps.
- The task is expected to run for 10 minutes or more.
- The task is generated by `/vibeguard:interview` or `/vibeguard:exec-plan` and will be resumed across sessions.

**Pinned surfaces**:

| Surface | Required baseline |
|------|------|
| Runtime version | Agent CLI version, model ID, and key SDK/runtime version |
| Tool list | Available tool, MCP server, and skill set, including description hashes when available |
| Rule set | Loaded VibeGuard rule files and their content hashes |

**Protocol**:
1. Before starting the long task, create a pinning snapshot and record its path in the plan, SPEC, or execution log.
2. Before resuming the task in a new session, or after installing/updating tools, run a drift check against the snapshot.
3. If drift is detected, stop and show the changed surface before any further task execution.
4. Continue only after the user explicitly accepts the drift or the environment is restored to the pinned snapshot.

**Mechanical checks (agent execution rules)**:
- Use `bash guards/universal/check_runtime_drift.sh --snapshot <file>` to create a baseline before long-task execution.
- Use `bash guards/universal/check_runtime_drift.sh --check <file>` before resumed execution or after tool/runtime changes.
- When the active agent runtime can export its loaded tool/MCP/skill list, pass it through `--tools-file <tool-dump>` so description hashes are pinned explicitly.
- A drift finding must name the changed surface: runtime, model, SDK/runtime, tool surface hash, or rule surface hash.
- Do not treat a clean final answer as proof that the intermediate environment was stable.

**Downgrade path**:
If the user accepts runtime drift, record the decision in a project-level decision log such as `SECURITY.md`, including the snapshot path, the changed surface, and why continuing is acceptable. The downgrade is task-scoped and must not become a global bypass.

**Relation to existing rules**:
- W-16 requires verification evidence from the current session; W-20 requires the execution environment behind that evidence to remain stable.
- W-14 handles spatial isolation between agents; W-20 handles temporal isolation across long tasks.
- SEC-12 and SEC-13 provide the MCP/high-context hash surfaces that W-20 consumes.
